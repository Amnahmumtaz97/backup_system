DataVault Secure Data Backup System
Detailed Project Report

Project Information
-------------------
Project Title: DataVault - Secure Data Backup System
Domain: Information Security
Technology Stack: Python, Flask, Cryptography, APScheduler, HTML/CSS/JavaScript
Project Type: Secure backup and restore application with web dashboard and CLI

1. Executive Summary
--------------------
DataVault is a secure, automated backup solution designed to protect user data against loss, tampering, and unauthorized access. The system supports Full, Incremental, and Differential backups, encrypts backup archives using AES-256, verifies archive integrity using SHA-256, and offers both a web dashboard and command-line interface for usability.

The project demonstrates practical implementation of information security principles through:
- Confidentiality via password-based encryption
- Integrity via hash-based tamper detection
- Availability via scheduled backups and restore capabilities

2. Problem Statement
--------------------
Users frequently store important files without a structured backup strategy. Common risks include:
- Data loss from accidental deletion or hardware failure
- Backup corruption without detection
- Unencrypted backups exposing sensitive information
- Complex tools that are difficult for non-technical users

DataVault addresses these risks by combining automated backup logic, strong cryptography, integrity validation, and an accessible dashboard.

3. Project Objectives
---------------------
Primary objectives:
1. Build a secure backup engine in Python
2. Support multiple backup strategies (Full, Incremental, Differential)
3. Encrypt archives with AES-256 using passphrase-derived keys
4. Detect tampering using SHA-256 archive verification
5. Provide reliable restore operations
6. Offer both web and CLI interfaces
7. Enable automatic scheduled backups
8. Maintain operation logs for auditing and troubleshooting

5. System Architecture
----------------------
DataVault uses a modular architecture with clear separation of responsibilities.

Core components:
1. Backup Engine (core/backup_engine.py)
   - Scans source files
   - Detects file changes
   - Creates archives
   - Handles encryption/decryption
   - Restores backups and chains
   - Verifies backup integrity

2. Manifest Manager (core/manifest.py)
   - Stores per-backup metadata as JSON
   - Retrieves latest and full backup references
   - Lists and deletes manifests

3. Scheduler (core/scheduler.py)
   - Uses APScheduler BackgroundScheduler
   - Supports interval-based automated backups
   - Persists schedule settings and status

4. Audit Logger (core/logger.py)
   - Writes timestamped log entries
   - Supports log retrieval for UI and CLI

5. Web Layer (app.py + ui/index.html)
   - REST API endpoints
   - Interactive dashboard for operations

6. CLI Layer (cli.py)
   - Scriptable command-line backup workflow

Data flow summary:
- User action (Web/CLI) -> BackupEngine -> archive + manifest + logs
- Restore action -> integrity check + decryption + extraction + chain replay

6. Security Design
------------------
6.1 Confidentiality
- Encryption algorithm: AES-256 in CBC mode
- Key derivation: PBKDF2-HMAC-SHA256 (100,000 iterations)
- Inputs: user passphrase + random per-backup salt
- Archive format includes metadata prefix for decryption compatibility
- Strong recommendation: use long unique passphrases

6.2 Integrity
- SHA-256 archive hash stored in backup manifest
- Hash is recomputed before restore
- Restore aborts if archive hash mismatch is detected
- Verification APIs support single and bulk integrity validation

6.3 Availability
- Full backup for complete snapshot
- Incremental backup for fast frequent updates
- Differential backup for balanced restore complexity
- Scheduled backups using APScheduler
- Chain-aware restore reconstructs required state

7. Backup Strategies Implemented
--------------------------------
7.1 Full Backup
- Captures all files from source directory
- Independent restore point
- Recommended for first backup and periodic baseline snapshots

7.2 Incremental Backup
- Captures only files changed since the last backup
- Fast and storage-efficient
- Restore requires replay of prior chain

7.3 Differential Backup
- Captures files changed since last full backup
- Slower than incremental but simpler restore chain

Deletion tracking:
- Deleted files are recorded in manifest metadata
- During chain restore, deleted files are removed from restored output

8. Restore and Recovery Model
-----------------------------
Restore process:
1. Resolve requested backup manifest
2. Build restore chain (Full -> Differential/Incremental sequence)
3. Verify hash for each archive in chain
4. Decrypt archive payload
5. Safely extract contents (path traversal checks)
6. Apply deleted-file operations from manifests
7. Return restore summary and execution duration

Error handling includes:
- Missing archive/manifests
- Integrity mismatch
- Decryption failure (wrong passphrase)
- Invalid/corrupt archive payload

14. Operational Guidelines
--------------------------
Recommended production usage pattern:
1. Configure a secure passphrase and source/destination paths
2. Run an initial Full backup
3. Enable periodic Incremental schedule
4. Run occasional Differential or Full backups
5. Verify backups regularly
6. Test restore process to a safe location
7. Monitor logs and storage usage

15. Risks and Mitigation
------------------------
Risk: Passphrase loss
Mitigation: Store passphrase in secure password manager; no plaintext sharing.

Risk: Local disk failure at destination
Mitigation: Copy encrypted archives to external/offsite media.

Risk: Misconfigured source path
Mitigation: Validate source directory before scheduling.

Risk: Corrupted archive
Mitigation: SHA-256 verification before restore and regular verify-all checks.
